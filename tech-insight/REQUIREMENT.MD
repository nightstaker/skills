
# Tech-Insight 项目需求文档 (Production-Ready v1.0)

## 1. 背景与目标 (Background & Goals)

### 1.1 背景
这是一个专业的**系统化技术洞察** Skill。当下 LLM 直接做技术洞察存在三大痛点：
* **不聚焦**：泛泛而谈"AI、大模型、芯片"，缺乏可落地的产品/技术抓手。
* **不系统**：观点散乱，缺少"场景—厂商—技术—商业"的递进推理链路。
* **不可信**：信息源不透明，关键技术细节凭"先验直觉"杜撰，无法被引用和复核。

本 Skill 的核心使命是把技术洞察从"灵感写作"升级为"工程化研究"——让每一个判断都可溯源、可复核、可推演。

### 1.2 核心目标
1. **系统化洞察**：以结构化框架（场景→厂商动态→关键技术→趋势预测→商业闭环）替代零散观点。
2. **可信溯源**：所有事实性陈述（产品规格、技术参数、厂商动作、市场数据）必须附带真实可访问的信息源。
3. **关键技术深挖**：对每一项被点名的关键技术，给出原理、实现路径、壁垒来源，并附上技术性原始资料（论文/专利/官方白皮书/技术博客）。
4. **结构化总结与启示**：每一个洞察点必须配套"Insight（事实）+ Implication（启示）"双层输出，支持后续战略/产品决策。
5. **闭环推演**：对未来趋势类洞察，强制输出"技术 → 产品 → 客户/应用场景 → 商业价值"的完整闭环。

## 2. 洞察分类定义 (Insight Taxonomy)

Tech-Insight 将所有洞察严格划分为四类，使用不同的分析框架与质量门槛。

### 2.1 类型 A：细分技术领域洞察 (Segmented Technology Domain Insight)
**目标**：针对某个限定的细分技术领域，从其应用发展入手，拆解分析最底层的需求，进一步转换为对该细分领域技术的需求；然后对该领域的 TOP 5 技术进行深入分析，按照"总—分—总"的结构，得出该细分领域的技术趋势和对该领域技术的研究启示。

**强制结构**：

1. **领域定义与应用发展（总）**：
   * **领域边界 (Domain Scope)**：明确该细分技术领域的定义、上下游关系、核心应用场景。
   * **应用发展脉络 (Application Evolution)**：梳理该领域应用的发展历程与当前态势，从应用演进中提炼出驱动技术发展的核心需求。
   * **底层需求拆解 (Fundamental Demand Decomposition)**：从应用需求逐层拆解到最底层的技术需求，建立"应用需求 → 系统需求 → 技术需求"的递进链路，每一层转换须有逻辑推导与事实支撑。

2. **TOP 5 技术深入分析（分）**：
   * **TOP 5 入选标准 (Selection Criteria)**：基于技术影响力、学术引用、产业采用率、成熟度等公开可核查指标选出该领域最具代表性的 5 项技术（或技术路线），至少 1 项指标带源；如领域过窄可降级为 TOP N（N≥3），须显式说明。
   * **逐项技术深入分析 (Per-Technology Deep Dive, ×5)**：对每项技术分别给出：
     - 技术原理与核心机制
     - 当前成熟度（TRL）与产业化程度
     - 关键性能指标与可量化对比（与其他技术路线横向对比）
     - 技术壁垒与护城河（专利、工艺、人才、生态）
     - 代表性成果/产品/论文（带源）

3. **技术趋势与研究启示（总）**：
   * **技术趋势归纳 (Technology Trend Synthesis)**：基于 TOP 5 技术分析，横向归纳该领域的共同演进方向、分歧点、技术收敛/发散趋势。
   * **研究启示 (Research Implication)**：对该领域技术研究的具体建议——应重点投入哪些技术方向、规避哪些已验证失败的路径、哪些交叉领域值得关注，回答"So what?"。

### 2.2 类型 B：行业技术趋势洞察 (Industry Technology Trend Insight)
**目标**：针对某个行业进行全面技术洞察，从行业发展史入手，引出主力的技术路线和有潜力的技术研究路线，再分别对各技术路线进行深入分析，最后总结并形成对该行业技术研究的建议。

**强制结构**：

1. **行业发展史与技术路线引出**：
   * **行业发展脉络 (Industry Evolution)**：梳理该行业从起源到当前的关键发展阶段、里程碑事件、市场规模演变，每条带源。
   * **技术路线图谱 (Technology Roadmap Landscape)**：从行业发展史中提炼出当前的主力技术路线（已被大规模产业化验证）和有潜力的技术研究路线（尚处实验/早期阶段但前景可期），明确分类依据。

2. **逐条技术路线深入分析 (Per-Route Deep Dive)**：
   * 对每条技术路线分别进行以下分析：
     - **成功产品引子（如有）**：若该技术路线已有成功产品，以最具代表性的 2–3 个产品作为引子，介绍其市场表现、客户反馈、核心卖点（带源）。
     - **技术优势与局限性 (Strengths & Limitations)**：系统性分析该技术路线的工程优势（性能、成本、可靠性等）和当前局限（瓶颈、缺陷、适用边界），均需可量化指标支撑。
     - **前沿研究方向 (Frontier Research)**：结合应用需求的变化趋势，讲解该技术路线的最新研究热点与突破方向，附论文/专利/白皮书等原始资料。

3. **全局总结与研究建议**：
   * **技术路线横向对比 (Cross-Route Comparison)**：以矩阵或对比表形式呈现各技术路线在关键维度上的优劣，明确哪条路线在何种场景下占优。
   * **行业技术研究建议 (Industry Research Recommendation)**：基于全部分析，给出对该行业技术研究的具体建议——主力路线的深耕方向、潜力路线的布局时机、跨路线融合的机会点，回答"So what?"。

### 2.3 类型 C：产品技术洞察 (Product Technology Insight)
**目标**：针对某个具体产品进行全面技术洞察，以该产品的卖点和特性入手，拆解其中的一个或多个关键技术架构，最后形成总结与对标产品的启示。

**强制结构**：

1. **产品概览与卖点分析**：
   * **产品定位 (Product Positioning)**：产品名称、厂商、目标客户、市场定位、上市时间、市场表现（份额/销量/营收），每条带源。
   * **核心卖点与特性 (Key Selling Points & Features)**：系统性梳理该产品对外宣传的核心卖点与差异化特性，区分"营销话术"与"可验证的工程事实"。

2. **关键技术架构拆解**：
   * **技术架构识别 (Architecture Identification)**：从卖点与特性中识别出支撑其差异化的 1–3 个关键技术架构。
   * **逐项技术架构拆解 (Per-Architecture Decomposition)**：对每个关键技术架构分别给出：
     - 系统架构与模块分解（附架构图/拆解图，带源）
     - 核心技术原理与实现路径
     - 关键性能参数与工程指标（带源）
     - 技术创新点与壁垒分析
     - 与同类产品的技术对比（如适用）

3. **总结与产品启示**：
   * **技术总结 (Technical Summary)**：归纳该产品的技术核心竞争力与技术短板。
   * **对标产品启示 (Product Implication)**：对我方或对标产品的具体启示——哪些技术值得借鉴、哪些设计可以超越、哪些路径应规避，回答"So what?"。

### 2.4 类型 D：公司技术洞察 (Company Technology Insight)
**目标**：针对某个公司进行全面技术洞察，覆盖其产品布局、公司战略、技术路线等，最后形成总结与对事业单元的启示。

**强制结构**：

1. **公司概览**：
   * **公司画像 (Company Profile)**：公司名称、成立时间、总部、核心业务领域、营收规模、员工规模、市值/估值，每条带源。
   * **发展历程与里程碑 (Evolution & Milestones)**：梳理公司关键发展节点、重大并购/投资、业务转型事件。

2. **产品布局分析**：
   * **产品矩阵 (Product Portfolio)**：系统性梳理公司当前的产品线/业务线布局，以矩阵或图谱形式呈现各产品的市场定位、目标客户、营收贡献（带源）。
   * **核心产品深挖 (Core Product Deep Dive)**：对 2–3 款核心产品进行详细分析——市场表现、竞争优势、技术亮点。

3. **公司战略与技术路线**：
   * **战略方向 (Strategic Direction)**：从公开资料（财报、投资者日、CEO 发言、专利布局、招聘动向）中提炼公司的技术战略方向，每条带源。
   * **技术路线图 (Technology Roadmap)**：梳理公司在核心技术领域的布局——自研 vs 收购 vs 合作，专利组合分析，研发投入趋势。
   * **技术能力评估 (Technical Capability Assessment)**：评估公司在关键技术领域的竞争力——人才储备、专利壁垒、工程化能力、生态位。

4. **总结与事业单元启示**：
   * **公司技术竞争力总结 (Competitiveness Summary)**：归纳该公司的技术核心优势、潜在风险与战略盲区。
   * **对事业单元的启示 (Business Unit Implication)**：对我方事业单元的具体建议——竞争策略、合作机会、技术差距弥补路径、差异化切入点，回答"So what?"。

## 3. 核心功能模块 (Function Modules)

### 3.1 信息源采集器 (Source Collector)
* **白名单优先**：一手资料优先级 > 二手资料，按 官方发布 / 专利 / 论文 / 监管文件 / 头部媒体评测 / 行业报告 / 个人博客 排序。
* **可访问性校验**：每一条引用必须给出可点击 URL 或可检索的标识符（DOI、专利号、SEC filing 编号）。
* **时效性约束**：厂商动态类信息默认窗口 12 个月，超过窗口必须显式标注"历史背景"。
* **多源交叉**：关键事实（参数、份额、价格）至少两个独立来源交叉验证，冲突时优先一手并标注分歧。
* **全量挂源 (Universal Sourcing)**：洞察报告中**任何**陈述句、数据、结论、引语、技术参数、厂商动作都必须就地附带信息源链接（URL / DOI / 专利号），禁止出现"无源句"。无法挂源的内容必须删除或显式标注为"作者推断"并独立分区。

### 3.1.1 视觉资料采集器 (Visual Asset Collector)
* **强制视觉证据**：洞察报告必须配套**原始视觉资料**——产品图、产品上下文图（使用场景、拆解图、PCB/Die shot）、技术架构图、数据流图、对比图表等，禁止纯文字洞察。
* **来源合法性**：图片必须来自一手资料（厂商官网、官方发布会、专利附图、论文 figure、监管文件、官方白皮书），并保留原始出处链接；不得使用无法溯源的二次加工图。
* **两种交付形态**：
  - **链接形态**：直接 inline 嵌入原始图片 URL（适合在线阅读）。
  - **下载形态**：把图片下载到 `assets/<slug>/` 目录，文件名带产品/章节前缀，并在 markdown 中相对引用；同时保留原始 URL 与抓取时间戳作为图注。
* **内容匹配 (Caption Binding)**：每张图必须紧邻其所支撑的论述段落，并以 `图 X：[说明] — 来源：[链接]` 的格式注明，严禁"配图与正文无关"。
* **技术架构图强制规则（全类型通用）**：所有技术洞察报告中，**每个技术点**必须至少配备 1 张技术架构图（原理图/系统架构图/数据流图/模块分解图），**每个技术点下涉及的每个具体产品/项目**也必须至少配备 1 张该产品的技术架构图（系统架构图/拆解图/Pipeline 图）。架构图必须来自一手资料（论文 figure、官方白皮书、专利附图、官方技术博客），禁止使用无法溯源的自绘图。
* **覆盖度要求（在技术架构图强制规则基础上的额外要求）**：类型 A 每项技术 ≥1 张原理/架构图 + 每项技术下的每个代表产品/项目 ≥1 张产品架构图；类型 B 每条技术路线 ≥1 张技术原理图 + 每条路线下的每个代表产品 ≥1 张产品架构图；类型 C 产品图 ≥1 张 + 每个关键架构 ≥1 张拆解图；类型 D 产品矩阵图 ≥1 张 + 每个核心产品 ≥1 张产品架构图。

### 3.2 关键技术拆解器 (Key Tech Decomposer)
* **强制下钻**：禁止在"使用了 Transformer / 采用 3nm 工艺"层级停止，必须下钻到具体架构选择、关键超参、工艺节点的差异化点。
* **可量化对比**：与上一代或主要竞品给出可量化的指标对比表（性能、功耗、成本、良率等），数据须带源。
* **壁垒分析**：识别该技术的护城河来源（专利、人才、制程、生态、数据），并评估可复制性。

### 3.3 总—分—总推演器 (Synthesis Reasoner)
* **结构完整性检查**：校验各类型的"总—分—总"或递进结构是否齐全——类型 A 的"需求拆解→技术分析→趋势归纳"、类型 B 的"行业史→技术路线分析→研究建议"、类型 C 的"卖点→架构拆解→启示"、类型 D 的"概览→战略分析→启示"，缺任一层拒绝输出。
* **假设显式化**：所有趋势预测或前瞻性判断必须列出 ≤5 条核心假设，并给出每条假设的可证伪条件。
* **横向对比锚点**：涉及多项技术或多条技术路线的分析必须给出可量化的横向对比表。

### 3.4 洞察质量校验器 (Insight Linter & QA)
* **可信度门槛**：每一个洞察点必须通过"事实-来源-启示"三段式校验，缺任一段直接打回。
* **全量挂源校验**：扫描全文，任何未挂载 URL/DOI/专利号的事实性陈述一律标红打回，零容忍。
* **视觉资料校验**：检查每节是否含原始图（产品图 / 架构图 / 上下文图），缺图直接打回；检查每张图是否带来源链接与"图—文匹配"的图注。
* **反幻觉检查**：识别未挂源的具体数字、产品代号、人物言论，强制补源或删除。
* **结构完整性**：类型 A 检查"领域定义与应用发展 / TOP 5 技术深入分析 / 技术趋势与研究启示"三层结构；类型 B 检查"行业发展史 / 逐条技术路线分析 / 全局总结与研究建议"三层结构；类型 C 检查"产品概览与卖点 / 关键技术架构拆解 / 总结与产品启示"三层结构；类型 D 检查"公司概览 / 产品布局与战略分析 / 总结与事业单元启示"三层结构。
* **启示可操作性**：Implication 必须回答"So what？"——对读者的决策/产品/投资有可执行的指向，禁止"需要持续关注"类空话。

## 5. 生产级技术指标 (Technical Specifications)

| 维度 | 要求说明 |
| :--- | :--- |
| **信息源等级** | 每个事实性陈述至少 1 条一手资料；关键参数至少 2 条独立来源交叉验证。 |
| **全量挂源** | 报告中任何陈述句、数据、引语均必须就地附 URL/DOI/专利号；无源内容必须删除或标注为"作者推断"并独立分区。 |
| **视觉证据** | 必须包含原始产品图、产品上下文图、技术架构图等；图片可为原始 URL 或下载至 `assets/<slug>/`，每张必须带源链接与"图—文匹配"图注。**全类型通用强制规则：每个技术点 ≥1 张技术架构图，每个技术点下的每个产品/项目 ≥1 张产品架构图**。额外要求：类型 A 每项技术 ≥1 张原理/架构图 + 每个代表产品 ≥1 张产品架构图；类型 B 每条技术路线 ≥1 张技术原理图 + 每个代表产品 ≥1 张产品架构图；类型 C 产品图 ≥1 张 + 每个关键架构 ≥1 张拆解图；类型 D 产品矩阵图 ≥1 张 + 每个核心产品 ≥1 张产品架构图。 |
| **时效窗口** | 厂商动态默认近 12 个月；超出窗口必须显式标注。 |
| **样本完整性** | 类型 A 必须覆盖该领域 TOP 5 技术（极端窄领域可降级 TOP N，N≥3），且入选依据带源；类型 B 必须覆盖主力与潜力技术路线；类型 D 必须覆盖 2–3 款核心产品。 |
| **结构完整性** | 类型 A 必含"领域定义与应用发展 / TOP 5 技术分析 / 技术趋势与研究启示"；类型 B 必含"行业发展史 / 逐条技术路线分析 / 全局总结与研究建议"；类型 C 必含"产品概览与卖点 / 关键技术架构拆解 / 总结与产品启示"；类型 D 必含"公司概览 / 产品布局与战略分析 / 总结与事业单元启示"。 |
| **下钻深度** | 关键技术拆解禁止停留在通用名词层级，必须给出可量化的差异化指标。 |
| **可证伪性** | 所有趋势预测必须附前提假设与可证伪条件，禁止无条件断言。 |
| **启示可操作性** | Implication 必须给出可执行的产品/战略/投资指向，禁止空泛表述。 |
| **反幻觉** | 任何具体数字、产品代号、引语未挂源即视为不合格，自动打回补源。 |
